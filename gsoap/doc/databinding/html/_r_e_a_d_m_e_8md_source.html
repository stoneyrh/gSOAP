2 XML Data Bindings {#mainpage}
3 =================
4 
5 [TOC]
6 
7 This is a detailed overview of the gSOAP XML data bindings concepts and
8 implementation. At the end of this document two examples are given to
9 illustrate the application of data bindings.
10 
11 The first example `address.cpp` shows how to use wsdl2h to bind an XML
12 schema to C++. The C++ application reads and writes an XML file into and from a
13 C++ "address book" data structure. The C++ data structure is an STL vector of
14 address objects.
15 
16 The second example `graph.cpp` shows how XML is serialized as a tree, digraph,
17 and cyclic graph. The digraph and cyclic graph serialization rules are similar
18 to SOAP 1.1/1.2 encoded multi-ref elements with id-ref attributes to link
19 elements through IDREF XML "pointers".
20 
21 These examples demonstrate XML data bindings only for relatively simple data
22 structures and types. The gSOAP tools support more than just these type of
23 structures, which we will explain in the next sections. Support for XML schema
24 components is practically unlimited. The wsdl2h tool maps schemas to C and C++
25 using built-in intuitive mapping rules, while allowing the mappings to be
26 customized using a `typemap.dat` file with mapping instructions for wsdl2h.
27 
28 The information in this document is applicable to gSOAP 2.8.24 and higher, which
29 supports C++11 features. However, C++11 is not required to use this material
30 and follow the example, unless we need smart pointers and scoped enumerations.
31 While most of the examples in this document are given in C++, the concepts also
32 apply to C with the exception of containers, smart pointers, classes and their
33 methods. None of these exceptions limit the use of the gSOAP tools for C in any
34 way.
35 
36 The data binding concepts described in this document have somewhat changed
37 and improved over the years since the first version of gSOAP was developed in
38 1999 (the project was called a "XML/SOAP stub/skeleton compiler" back
39 then). However, the principle of mapping XSD components to C/C++ types and vice
40 versa was envisioned, designed, and implemented early on in research conducted
41 by Dr. van Engelen at the Florida State University and subsequently adopted by
42 other tools, including Java web services and in C# WCF.

45 Mapping WSDL and XML schemas to C/C++ {#tocpp}
46 =====================================
47 
48 To convert WSDL and XML schemas (XSD files), we use the wsdl2h command to
49 generate the data binding interface code in a special gSOAP header file:
50 
51  wsdl2h [options] -o file.h ... XSD and WSDL files ...
52 
53 This converts WSDL and XSD files to C++ (or pure C with wsdl2h option `-c`) and
54 saves a special `file.h` data binding interface file.
55 
56 The WSDL 1.1/2.0, SOAP 1.1/1.2, and XSD 1.0/1.1 standards are supported by the
57 gSOAP tools. In addition, the most popular WS specifications are also
58 supported, including WS-Addressing, WS-ReliableMessaging, WS-Discovery,
59 WS-Security, WS-Policy, WS-SecurityPolicy, and WS-SecureConversation.
60 
61 This document focusses on XML data bindings and mapping C/C++ to XML 1.0/1.1
62 and XSD 1.0/1.1. This covers all of the following standard XSD components with
63 their optional `[ attributes ]` properties:
64 
65  any [minOccurs, maxOccurs]
66  anyAttribute
67  all
68  choice [minOccurs, maxOccurs]
69  sequence [minOccurs, maxOccurs]
70  group [name, ref]
71  attributeGroup [name, ref]
72  attribute [name, ref, type, use, default, fixed, form, wsdl:arrayType]
73  element [name, ref, type, default, fixed, form, nillable, abstract, substitutionGroup, minOccurs, maxOccurs]
74  simpleType [name]
75  complexType [name, abstract, mixed]
76 
77 And also the following standard XSD components:
78 
79  import imports a schema into the importing schema for referencing
80  include include schema component definitions into a schema
81  override override by replacing schema component definitions
82  redefine extend or restrict schema component definitions
83  annotation annotates a component
84

86 
87  enumeration maps to enum
88  simpleContent maps to class/struct wrapper with __item member
89  complexContent maps to class/struct
90  list maps to enum* bitmask (enum* enumerates up to 64 bit masks)
91  extension through inheritance
92  restriction partly through inheritance and redeclaration
93  length restricts content length
94  minLength restricst content length
95  maxLength restricst content length
96  minInclusive restricts numerical value range
97  maxInclusive restricts numerical value range
98  minExclusive restricts numerical value range
99  maxExclusive restricts numerical value range
100  precision maps to float/double but constraint is not validated
101  scale maps to float/double but constraint is not validated
102  totalDigits maps to float/double but constraint is not validated
103  fractionDigits maps to float/double but constraint is not validated
104  pattern must define `soap::fsvalidate` callback to validate patterns
105  union maps to string of values

109 
110  anyType maps to _XML string with literal XML content (or DOM with wsdl2h option -d)

112  string maps to string (char*/wchar_t*/std::string/std::wstring)
113  boolean maps to bool (C++) or enum xsd__boolean (C)
114  byte maps to char (int8_t)
115  short maps to short (int16_t)
116  int maps to int (int32_t)
117  long maps to LONG64 (long long and int64_t)
118  unsignedByte maps to unsigned char (uint8_t)
119  unsignedShort maps to unsigned short (uint16_t)
120  unsignedInt maps to unsigned int (uint32_t)
121  unsignedLong maps to ULONG64 (unsigned long long and uint64_t)
122  float maps to float
123  double maps to double
124  integer maps to string
125  decimal maps to string, or use "#import "custom/long_double.h"
126  precisionDecimal maps to string
127  duration maps to string, or use "#import "custom/duration.h"

136 generating a typedef. For example, xsd:token is bound to a C++ or C string,
137 which associates a value space to the type with the appropriate XSD type name
138 used by the soapcpp2-generated serializers:
139 
140  typedef std::string xsd__token; // C++
141  typedef char *xsd__token; // C (wsdl2h option -c)

147 Using typemap.dat to customize generated bindings {#typemap}
148 =================================================
149 
150 We use a `typemap.dat` file to redefine namespace prefixes and to customize
151 type bindings for the the generated header files produced by the wsdl2h tool.
152 The `typemap.dat` is the default file processed by wsdl2h. Use wsdl2h option
153 `-t` to specify an alternate file.
154 
155 Declarations in `typemap.dat` can be broken up over multiple lines by
156 continuing on the next line by ending each line to be continued with a
157 backslash `\`.
158 
159 XML namespace bindings {#typemap1}
160 ----------------------
161 
162 The wsdl2h tool generates C/C++ type declarations that use `ns1`, `ns2`, etc.
163 as URI schema-binding prefixes. These default prefixes are generated somewhat
164 arbitrarily for each schema URI, meaning that their ordering may change
165 depending on the WSDL and XSD order of processing with wsdl2h.
166 
167 It is **strongly recommended** to declare your own prefix for each schema URI
168 to enhance maintaince of your code. This is to anticipate possible changes of
169 the schema(s) and/or the binding URI(s) and/or the tooling procedures.
170 
171 Therefore, the first and foremost important thing to do is to define prefix-URI
172 bindings for our C/C++ code by adding the following line(s) to our
173 `typemap.dat` or make a copy of this file and add the line(s) that bind our
174 choice of prefix name to each URI:
175 
176  prefix = "URI"
177 
178 For example:
179 
180  g = "urn:graph"
181 
182 This produces `g__name` C/C++ type names that are bound to the "urn:graph"
183 schema by association of `g` to the C/C++ types.
184 
185 This means that `<g:name xmlns:g="urn:graph">` is parsed as an instance of a
186 `g__name` C/C++ type. Also `<x:name xmlns:x="urn:graph">` parses as an instance
187 of `g__name`, because the prefix `x` has the same URI value `urn:graph`.
188 Prefixes in XML have local scopes (like variables in a block).
189 
190 The first run of wsdl2h will reveal the URIs, so we do not need to search WSDLs
191 and XSD files for all of the target namespaces.
192 
193 XSD type bindings {#typemap2}
194 -----------------
195 
196 Custom C/C++ type bindings can be declared in `typemap.dat` to associate C/C++
197 types with specific schema types. These type bindings have four parts:
198 
199  prefix__type = declaration | use | ptruse
200 
201 where
202 
203 - `prefix__type` is the schema type to be customized (the `prefix__type` name
204  uses the common double underscore naming convention);
205 - `declaration` declares the C/C++ type in the wsdl2h-generated header file.
206  This part can be empty if no explicit declaration is needed;
207 - `use` is an optional part that specifies how the C/C++ type is used in the
208  code. When omitted, it is the same as `prefix__type`;
209 - `ptruse` is an optional part that specifies how the type is used as a
210  pointer type. By default it is the `use` type name with a `*` or C++11
211  `std::shared_ptr<>` (see further below).
212 
213 For example, to map xsd:duration to a `long long` (`LONG64`) milliseconds
214 value, we can use the custom serializer declared in `custom/duration.h` by
215 adding the following line to `typemap.dat`:
216 
217  xsd__duration = #import "custom/duration.h" | xsd__duration
218 
219 Here, we could have omitted the second field, because `xsd__duration` is the
220 name that wsdl2h uses to identify this type.
221 
222 To map xsd:string to `wchar_t*` wide strings:
223 
224  xsd__string = | wchar_t* | wchar_t*
225 
226 Note that the first field is empty, because `wchar_t` is a C type and does not
227 need to be declared. A `ptruse` field is given so that we do not end up
228 generating the wrong pointer types, such as `wchar_t**` and
229 `std::shared_ptr<wchar_t>`.
230 
231 When the auto-generated declaration should be preserved but the `use` or
232 `ptruse` fields replaced, then we use an ellipsis for the declaration part:
233 
234  prefix__type = ... | use | ptruse
235 
236 This is useful to map schema polymorphic types to C types for example, where we
237 need to be able to both handle a base type and its extensions as per schema
238 extensibility. Say we have a base type called ns:base that is extended, then we
239 can remap this to a C type that permits referening the extended types via a
240 `void*` as follows:
241 
242  ns__base = ... | int __type_base; void*
243 
244 such that `__type_base` and `void*` are used to (de)serialize any data type,
245 including base and its derived types.
246 
247 Class/struct member additions {#typemap3}
248 -----------------------------
249 
250 All generated classes and structs can be augmented with additional
251 members such as methods, constructors and destructors, and private members:
252 
253  prefix__type = $ member-declaration
254 
255 For example, we can add method declarations and private members to a class, say
256 `ns__record` as follows:
257 
258  ns__record = $ ns__record(const ns__record &); // copy constructor
259  ns__record = $ void print(); // a print method
260  ns__record = $ private: int status; // a private member
261 
262 Note that method declarations cannot include any code, because soapcpp2's input
263 permits only type declarations, not code.
264 
265 Replacing XSD types by equivalent alternatives {#typemap4}
266 ----------------------------------------------
267 
268 Type replacements can be given to replace one type entirely with another given
269 type:
270 
271  prefix__type1 == prefix__type2
272 
273 This replaces all `prefix__type1` by `prefix__type2` in the wsdl2h output.
274 However, care muse be taken not to agressively replace types, because this can
275 cause XML validation to fail when a value-type mismatch is encountered in the
276 XML input. Therefore, only replace similar types with other similar types that
277 are wider (e.g. `short` by `int` and `float` by `double`).
278 
279 The built-in typemap.dat variables $CONTAINER and $POINTER {#typemap5}
280 ----------------------------------------------------------
281 
282 The `typemap.dat` `$CONTAINER` variable defines the container to emit in the
283 generated declarations, which is `std::vector` by default. For example, to emit
284 `std::list` as the container in the wsdl2h-generated declarations:
285 
286  $CONTAINER = std::list
287 
288 The `typemap.dat` `$POINTER` variable defines the smart pointer to emit in the
289 generated declarations, which replaces the use of `*` pointers. For example:
290 
291  $POINTER = std::shared_ptr
292 
293 Not all pointers in the generated output can be replaced by smart pointers.
294 Standard pointers are itill sed as union members and for pointers to arrays of
295 objects.
296 
297 User-defined content {#typemap6}
298 --------------------
299 
300 Any other content to be generated by wsdl2h can be included in `typemap.dat` by
301 enclosing it within brackets `[` and `]` anywhere in the `typemap.dat` file.
302 Each of the two brackets MUST appear at the start of a new line.
303 
304 For example, we can add an `#import "wsa5.h"` directive to the wsdl2h-generated
305 output as follows:
306 
307  [
308  #import "wsa5.h"
309  ]
310 
311 which emits the `#import "wsa5.h"` literally at the start of the
312 wsdl2h-generated header file.

315 Mapping C/C++ to XML schema {#toxsd}
316 ===========================
317 
318 The soapcpp2 command generates the data binding implementation code from a data
319 binding interface `file.h`:
320 
321  soapcpp2 [options] file.h
322 
323 where `file.h` is a gSOAP header file that declares the XML data binding
324 interface. The `file.h` is typically generated by wsdl2h, but we can also
325 declare one ourself. If so, we add gSOAP directives and declare in this file
326 all our C/C++ types we want to serialize in XML. We can also declare functions
327 that will be converted to service operations by soapcpp2.
328 
329 Global function declarations define service operations, which are of the form:
330 
331  int ns__name(arg1, arg2, ..., argn, result);
332 
333 where `arg1`, `arg2`, ..., `argn` are formal argument declarations of the input
334 and `result` is a formal argument for the output, which must be a pointer or
335 reference to the result object to be populated. More information can be found
336 in the gSOAP user guide.

434 Colon notation versus name prefixing {#toxsd2}
435 ------------------------------------
436 
437 To bind C/C++ type names to XSD types, a simple form of name prefixing is used
438 by the gSOAP tools by prepending the XML namespace prefix to the C/C++ type
439 name with a pair of undescrores. This also ensures that name clashes cannot
440 occur when multiple WSDL and XSD files are converted to C/C++. Also, C++
441 namespaces are not sufficiently rich to capture XML schema namespaces
442 accurately, for example when class members are associated with schema elements
443 defined in another XML namespace and thus the XML namespace scope of the
444 member's name is relevant, not just its type.
445 
446 However, from a C/C++ centric point of view this can be cumbersome. Therefore,
447 colon notation is an alternative to physically augmenting C/C++ names with
448 prefixes.
449 
450 For example, the following class uses colon notation to bind the `record` class
451 to the `urn:types` schema:
452 
453  //gsoap ns schema namespace: urn:types
454  class ns:record // binding 'ns:' to a type name
455  {
456  public:
457  std::string name;
458  uint64_t SSN;
459  ns:record *spouse; // using 'ns:' with the type name
460  ns:record(); // using 'ns:' here too
461  ~ns:record(); // and here
462  };
463 
464 The colon notation is stripped away by soapcpp2 when generating the data
465 binding implementation code for our project. So the final code just uses
466 `record` to identify this class and its constructor/destructor.
467 
468 When using colon notation we have to be consistent as we cannot use both forms
469 together. That is, `ns:record` differs from `ns__record` as a name.
470 
471 C++ Bool and C alternative {#toxsd3}
472 --------------------------
473 
474 The C++ `bool` type is bound to built-in XSD type xsd:boolean.
475 
476 The C alternative is to define an enumeration:
477 
478  enum xsd__boolean { false_, true_ };
479 
480 or by defining an enumeration in C with pseudo-scoped enumeration values:
481 
482  enum xsd__boolean { xsd__boolean__false, xsd__boolean__true };
483 
484 The XML value space of these types is `false` and `true`, but also accepts `0`
485 and `1` as values.
486 
487 To prevent name clashes, `false_` and `true_` have an underscore which are
488 removed in the XML value space.
489 
490 Enumerations and bitmasks {#toxsd4}
491 -------------------------
492 
493 Enumerations are mapped to XSD simpleType enumeration restrictions of
494 xsd:string, xsd:QName, and xsd:long.
495 
496 Consider for example:
497 
498  enum ns__Color { RED, WHITE, BLUE };
499 
500 which maps to a simpleType restriction of xsd:string in the soapcpp2-generated
501 schema:
502 
503  <simpleType name="Color">
504  <restriction base="xsd:string">
505  <enumeration value="RED"/>
506  <enumeration value="WHITE"/>
507  <enumeration value="BLUE"/>
508  </restriction>
509  </simpleType>
510 
511 Enumeration name constants can be pseudo-scoped to prevent name clashes,
512 because enumeration name constants have a global scope in C and C++:
513 
514  enum ns__Color { ns__Color__RED, ns__Color__WHITE, ns__Color__BLUE };
515 
516 We can also use C++11 scoped enumerations to prevent name clashes:
517 
518  enum class ns__Color : int { RED, WHITE, BLUE };
519 
520 Here, the type part `: int` is optional. In place of `int` in the example
521 above, we can also use `int8_t`, `int16_t`, `int32_t`, or `int64_t`.
522 
523 The XML value space of the enumertions defined above is `RED`, `WHITE`, and
524 `BLUE`.
525 
526 Prefix-qualified enumeration name constants are mapped to simpleType
527 restrictions of xsd:QName, for example:
528 
529  enum ns__types { xsd__int, xsd__float };
530 
531 which maps to a simpleType restriction of xsd:QName in the soapcpp2-generated
532 schema:
533 
534  <simpleType name="types">
535  <restriction base="xsd:QName">
536  <enumeration value="xsd:int"/>
537  <enumeration value="xsd:float"/>
538  </restriction>
539  </simpleType>
540 
541 Enumeration name constants can be pseudo-numeric as follows:
542 
543  enum ns__Primes { _3 = 3, _5 = 5, _7 = 7, _11 = 11 };
544 
545 which maps to a simpleType restriction of `xsd:long`:
546 
547  <simpleType name="Color">
548  <restriction base="xsd:long">
549  <enumeration value="3"/>
550  <enumeration value="5"/>
551  <enumeration value="7"/>
552  <enumeration value="11"/>
553  </restriction>
554  </simpleType>
555 
556 The XML value space of this type is `3`, `5`, `7`, and `11`.
557 
558 Besides (pseudo-) scoped enumerations, another way to prevent name clashes
559 accross enumerations is to start an enumeration name constant with one
560 underscore or followed it by any number of underscores, which makes it
561 unique. The leading and trailing underscores are removed in the XML value
562 space.
563 
564  enum ns__ABC { A, B, C };
565  enum ns__BA { B, A }; // BAD: B = 1 but B is already defined as 2
566  enum ns__BA_ { B_, A_ }; // OK
567 
568 The gSOAP soapcpp2 tool permits reusing enumeration name constants in other
569 (non-scoped) enumerations as long as these values are assigned the same
570 constant. Therefore, the following is permitted:
571 
572  enum ns__Primes { _3 = 3, _5 = 5, _7 = 7, _11 = 11 };
573  enum ns__Throws { _1 = 1, _2 = 2, _3 = 3, _4 = 4, _5 = 5, _6 = 6 };
574 
575 A bitmask type is an `enum*` "product" enumeration with a geometric,
576 power-of-two sequence of values assigned to the name constants:
577 
578  enum* ns__Options { SSL3, TLS10, TLS11, TLS12 };
579 
580 where the product enum assigns 1 to `SSL3`, 2 to `TLS10`, 4 to `TLS11`, and 8
581 to `TLS12`, which allows the enumeration values to be used in composing
582 bitmasks with `|` (bitwise or) `&` (bitwise and), and `~` (bitwise not):
583 
584  enum ns__Options options = (enum ns__Options)(SSL3 | TLS10 | TLS11 | TLS12);
585  if (options & SSL3) // if SSL3 is an option, warn and remove from options
586  {
587  warning();
588  options &= ~SSL3;
589  }
590 
591 The bitmask type maps to a simpleType list restriction of xsd:string in the
592 soapcpp2-generated schema:
593 
594  <simpleType name="Options">
595  <list>
596  <restriction base="xsd:string">
597  <enumeration value="SSL3"/>
598  <enumeration value="TLS10"/>
599  <enumeration value="TLS11"/>
600  <enumeration value="TLS12"/>
601  </restriction>
602  </list>
603  </simpleType>
604 
605 The XML value space of this type consists of all 16 possible subsets of the
606 four values, represented by an XML string with space-separated values. For
607 example, the bitmask `TLS10 | TLS11 | TLS12` equals 14 and is represented in by
608 the XML string `TLS10 TLS11 TLS12`.
609 
610 To convert `enum` name constants to string, we use the soapcpp2 auto-generated
611 `const char *soap_T2s(soap, enum T)` function.
612 
613 To convert a string to an `enum` name constant, we use the soapcpp2
614 auto-generated `int soap_s2T(soap, const char *str, enum T*)` function.
615 
616 Numerical types {#toxsd5}
617 ---------------
618 
619 Integer and floating point types are mapped to the equivalent built-in XSD
620 types with the same sign and bit width.
621 
622 The `size_t` type is transient (not serializable) because its width is platform
623 dependent. We recommend to use `uint64_t` instead.
624 
625 The XML value space of integer types are their decimal representations without
626 loss of precision.
627 
628 The XML value space of floating point types are their decimal representations.
629 The decimal representations are formatted with the printf format string "%.9G"
630 for floats and the printf format string "%.17lG" for double. The value space
631 includes the values `INF`, `-INF`, and `NAN`. To change the format string, we
632 can change one of these `struct soap` context data members:
633 
634  const char * soap::float_format
635  const char * soap::double_format
636 
637 Note that decimal conversions may result in a loss of precision of the least
638 significant decimal.
639 
640 A `long double` 128 bit floating point value requires a custom serializer:
641 
642  #import "custom/long_double.h"
643  typedef long double xsd__decimal;
644 
645 Compile and link your code with `custom/long_double.c`.
646 
647 The range of a numerical type can be restricted with a typedef:
648 
649  typedef int ns__narrow -10:10;
650 
651 which maps to a simpleType restriction of xsd:int in the soapcpp2-generated
652 schema:
653 
654  <simpleType name="narrow">
655  <restriction base="xsd:int">
656  <minInclusive value="-10"/>
657  <maxInclusive value="10"/>
658  </restriction>
659  </simpleType>
660 
661 The range of a float type can only be restricted within integral bounds. This
662 restriction may be dropped in future releases.
663 
664 String types {#toxsd6}
665 ------------
666 
667 String types are mapped to the built-in xsd:string and xsd:QName XSD types.
668 
669 The wide strings `wchar_t*` and `std::wstring` may contain Unicode that is
670 preserved in the XML value space.
671 
672 Strings `char*` and `std::string` can only contain extended Latin, but we can
673 store UTF-8 content that is preserved in the XML value space when the `struct
674 soap` context is initialized with the flag `XML_C_UTFSTRING`.
675 
676 Beware that many XML 1.0 parsers reject all control characters (those between
677 `#x1` and `#x1F`) except `#x9`, `#xA`, and `#xD`. With the newer XML 1.1
678 parsers (including gSOAP) you should be fine.
679 
680 The length of a string type can be restricted with a typedef:
681 
682  typedef std::string ns__password 6:16;
683 
684 which maps to a simpleType restriction of xsd:string in the soapcpp2-generated
685 schema:
686 
687  <simpleType name="password">
688  <restriction base="xsd:string">
689  <minLength value="6"/>
690  <maxLength value="16"/>
691  </restriction>
692  </simpleType>
693 
694 In addition, an XSD regex pattern restriction can be associated with a string
695 typedef:
696 
697  typedef std::string ns__password "([a-zA-Z]|[0-9]|-)+" 6:16;
698 
699 which maps to a simpleType restriction of xsd:string in the soapcpp2-generated
700 schema:
701 
702  <simpleType name="password">
703  <restriction base="xsd:string">
704  <pattern value="([a-zA-Z0-9]|-)+"/>
705  <minLength value="6"/>
706  <maxLength value="16"/>
707  </restriction>
708  </simpleType>
709 
710 Fixed-size strings (`char[N]`) are rare occurrences in the wild, but apparently
711 still used in some projects to store strings. To facilitate fixed-size string
712 serialization, use soapcpp2 option `-b`:
713 
714  typedef char ns__buffer[10]; // requires soapcpp2 option -b
715 
716 which maps to a simpleType restriction of xsd:string in the soapcpp2-generated
717 schema:
718 
719  <simpleType name="buffer">
720  <restriction base="xsd:string">
721  <maxLength value="9"/>
722  </restriction>
723  </simpleType>
724 
725 Note that fixed-size strings MUST contain NUL-terminated text and SHOULD NOT
726 contain raw binary data. Also, the length limitation is more restrictive for
727 UTF-8 content (enabled with the `SOAP_C_UTFSTRING`) that requires multibyte
728 character encodings. As a consequence, UTF-8 content may be truncated to fit.
729 
730 Note that raw binary data can be stored in a `xsd__base64Binary` or
731 `xsd__hexBinary` structure, or transmitted as a MIME attachment.
732 
733 The built-in `_QName` type is a regular C string type (`char*`) that maps to
734 xsd:QName but has the added advantage that it holds normalized qualified names.
735 There are actually two forms of normalized QName content, to ensure any QName
736 is represented accurately and uniquely:
737 
738  prefix:name
739  "URI":name
740 
741 where the first form is used when the prefix (and the binding URI) is defined
742 in the namespace table and is bound to a URI (see the .nsmap file). The second
743 form is used when the URI is not defined in the namespace table and therefore
744 no prefix is available to bind and normalize the URI to.
745 
746 A `_QName` string may contain a sequence of space-separated QName values, not
747 just one, and all QName values are normalized to the format shown above.
748 
749 To define a `std::string` base type for xsd:QName, we use a typedef:
750 
751  typedef std::string xsd__QName;
752 
753 The `xsd__QName` string content is normalized, just as with the `_QName`
754 normalization.
755 
756 To serialize strings that contain literal XML content to be reproduced in the
757 XML value space, use the built-in `_XML` string type, which is a regular C
758 string type (`char*`) that maps to plain XML CDATA.
759 
760 To define a `std::string` base type for literal XML content, use a typedef:
761 
762  typedef std::string XML;
763 
764 Strings can hold any of the values of the XSD built-in primitive types. We can
765 use a string typedef to declare the use of the string type as a XSD built-in
766 type:
767 
768  typedef std::string xsd__token;
769 
770 We MUST ensure that the string values we populate in this type conform to the
771 XML standard, which in case of xsd:token is: the lexical and value spaces of
772 xsd:token are the sets of all strings after whitespace replacement of any
773 occurrence of `#x9`, `#xA` , and `#xD` by `#x20` and collapsing.
774 
775 Date and time types {#toxsd7}
776 -------------------
777 
778 The C/C++ `time_t` type is mapped to the built-in xsd:dateTime XSD type that
779 represents a date and time within a time zone (typically UTC).
780 
781 The XML value space contains ISO 8601 Gregorian time instances of the form
782 `[-]CCYY-MM-DDThh:mm:ss.sss[Z|(+|-)hh:mm]`, where `Z` is the UTC time zone or a
783 time zone offset `(+|-)hh:mm]` from UTC is used.
784 
785 A `time_t` value is considered and represented in UTC by the serializer.
786 
787 Because the `time_t` value range is restricted to dates after 01/01/1970, care
788 must be taken to ensure the range of xsd:dateTime values in XML exchanges do
789 not exceed the `time_t` range.
790 
791 This restriction does not hold for `struct tm` (`<time.h>`), which we can use
792 to store and communicate a date and time in UTC without date range restrictions.
793 The serializer uses the `tm` data members directly for conversion to/from the
794 XML value space of xsd:dateTime:
795 
796  struct tm
797  {
798  int tm_sec; // seconds (0 - 60)
799  int tm_min; // minutes (0 - 59)
800  int tm_hour; // hours (0 - 23)
801  int tm_mday; // day of month (1 - 31)
802  int tm_mon; // month of year (0 - 11)
803  int tm_year; // year - 1900
804  int tm_wday; // day of week (Sunday = 0) (NOT USED)
805  int tm_yday; // day of year (0 - 365) (NOT USED)
806  int tm_isdst; // is summer time in effect?
807  char* tm_zone; // abbreviation of timezone (NOT USED)
808  };
809 
810 The `struct tm` type is mapped to the built-in xsd:dateTime XSD type and
811 serialized with the custom serializer `custom/struct_tm.h` that declares a
812 `xsd__dateTime` type:
813 
814  #import "custom/struct_tm.h" // import typedef struct tm xsd__dateTime;
815  ... use xsd__dateTime ...
816 
817 Compile and link your code with `custom/struct_tm.c`.
818 
819 The `struct tm` type is mapped to the built-in xsd:date XSD type and serialized
820 with the custom serializer `custom/struct_tm_date.h` that declares a
821 `xsd__date` type:
822 
823  #import "custom/struct_tm_date.h" // import typedef struct tm xsd__date;
824  ... use xsd__date ...
825 
826 Compile and link your code with `custom/struct_tm_date.c`.
827 
828 The XML value space of xsd:date are Gregorian calendar dates of the form
829 `[-]CCYY-MM-DD[Z|(+|-)hh:mm]`.
830 
831 The `struct timeval` (`<sys/time.h>`) type is mapped to the built-in
832 xsd:dateTime XSD type and serialized with the custom serializer
833 `custom/struct_timeval.h` that declares a `xsd__dateTime` type:
834 
835  #import "custom/struct_timeval.h" // import typedef struct timeval xsd__dateTime;
836  ... use xsd__dateTime ...
837 
838 Compile and link your code with `custom/struct_timeval.c`.
839 
840 Note that the same value range restrictions apply to `struct timeval` as they
841 apply to `time_t`. The added benefit of `struct timeval` is the addition of
842 a microsecond-precise clock:
843 
844  struct timeval
845  {
846  time_t tv_sec; // seconds since Jan. 1, 1970
847  suseconds_t tv_usec; // and microseconds
848  };
849 
850 An `unsigned long long` (`ULONG64` or `uint64_t`) type that contains a 24 hour
851 time in microseconds UTC is mapped to the built-in xsd:time XSD type and
852 serialized with the custom serializer `custom/long_time.h` that declares a
853 `xsd__time` type:
854 
855  #import "custom/long_time.h" // import typedef unsigned long long xsd__time;
856  ... use xsd__time ...
857 
858 Compile and link your code with `custom/long_time.c`.
859 
860 The XML value space of xsd:time are points in time recurring each day of the
861 form `hh:mm:ss.sss[Z|(+|-)hh:mm]`, where `Z` is the UTC time zone or a time
862 zone offset from UTC is used. The `xsd__time` value is always considered and
863 represented in UTC by the serializer.
864 
865 A C++11 `std::chrono::system_clock::time_point` type is mapped to the built-in
866 xsd:dateTime XSD type and serialized with the custom serializer
867 `custom/chrono_time_point.h` that declares a `xsd__dateTime` type:
868 
869  #import "custom/chrono_time_point.h" // import typedef std::chrono::system_clock::time_point xsd__dateTime;
870  ... use xsd__dateTime ...
871 
872 Compile and link your code with `custom/chrono_time_point.cpp`.

895 XSD type and serialized with the custom serializer `custom/chrono_duration.h`

903 Classes and structs {#toxsd9}
904 -------------------
905 
906 Classes and structs are mapped to XSD complexTypes. The XML value space
907 consists of XML elements with attributes and subelements, possibly constrained
908 by validation rules that enforce element and attribute occurrence contraints,
909 numerical value range constraints, and string length and pattern constraints.
910 
911 Classes that are declared with the gSOAP tools are limited to single
912 inheritence only. Structs cannot be inherited.
913 
914 The class and struct name is bound to an XML namespace by means of the prefix
915 naming convention or by using [colon notation](#toxsd1):
916 
917  //gsoap ns schema namespace: urn:types
918  class ns__record
919  {
920  public:
921  std::string name;
922  uint64_t SSN;
923  ns__record *spouse;
924  ns__record();
925  ~ns__record();
926  protected:
927  struct soap *soap;
928  };
929 
930 In the example above, we also added a context pointer to the `struct soap` that
931 manages this instance. It is set when the instance is created in the engine's
932 context, for example when deserialized and populated by the engine.
933 
934 The class maps to a complexType in the soapcpp2-generated schema:
935 
936  <complexType name="record">
937  <sequence>
938  <element name="name" type="xsd:string" minOccurs="1" maxOccurs="1"/>
939  <element name="SSN" type="xsd:unsignedLong" minOccurs="1" maxOccurs="1"/>
940  <element name="spouse" type="ns:record" minOccurs="0" maxOccurs="1" nillable="true"/>
941  </sequence>
942  </complexType>
943 
944 ### Serializable versus transient types and members
945 
946 Public data members of a class or struct are serialized. Private and protected
947 members are transient and not serializable.
948 
949 Also `const` and `static` members are not serializable, with the exception of
950 `const char*` and `const wchar_t*`.
951 
952 Types and specific class/struct members can be made transient by using the
953 `extern` qualifier:
954 
955  extern class std::ostream; // declare 'std::ostream' transient
956  class ns__record
957  {
958  public:
959  extern int num; // not serialized
960  std::ostream out; // not serialized
961  static const int MAX = 1024; // not serialized
962  };
963 
964 By declaring `std::ostream` transient we can use this type where we need it and
965 without soapcpp2 complaining that this class is not defined.
966 
967 ### Volatile classes and structs
968 
969 Classes and structs can be declared `volatile` with the gSOAP tools. This means
970 that they are already declared elsewhere in our project's source code. We do
971 not want soapcpp2 to generate a second definition for these types.
972 
973 For example, `struct tm` is declared in `<time.h>`. We want it serializable and
974 serialize only a selection of its data members:
975 
976  volatile struct tm
977  {
978  int tm_sec; // seconds (0 - 60)
979  int tm_min; // minutes (0 - 59)
980  int tm_hour; // hours (0 - 23)
981  int tm_mday; // day of month (1 - 31)
982  int tm_mon; // month of year (0 - 11)
983  int tm_year; // year - 1900
984  };
985 
986 We can declare classes and structs `volatile` for any such types we want to
987 serialize by only providing the public data members we want to serialize.
988 
989 Colon notation comes in handy to bind an existing class or struct to a schema.
990 For example, we can change the `tm` name as follows without affecting the code
991 that uses `struct tm` generated by soapcpp2:
992 
993  volatile struct ns:tm { ... }
994 
995 This struct maps to a complexType in the soapcpp2-generated schema:
996 
997  <complexType name="tm">
998  <sequence>
999  <element name="tm-sec" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1000  <element name="tm-min" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1001  <element name="tm-hour" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1002  <element name="tm-mday" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1003  <element name="tm-mon" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1004  <element name="tm-year" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1005  </sequence>
1006  </complexType>
1007 
1008 ### Mutable classes and structs
1009 
1010 Classes and structs can be declared `mutable` with the gSOAP tools. This means
1011 that their definition can be spread out over the source code. This promotes the
1012 concept of a class or struct as a *row of named values*, also known as a *named
1013 tuple*, that can be extended as needed with additional entries. Because these
1014 types differ from the traditional object-oriented principles of classes and
1015 objects, constructors and destructors cannot be defined (also because we cannot
1016 guarantee merging these into one such that all members will be initialized). A
1017 default constructor, copy constructor, assignment operation, and destructor
1018 will be assigned.
1019 
1020  mutable struct ns__tuple
1021  {
1022  @std::string id;
1023  };
1024 
1025  mutable struct ns__tuple
1026  {
1027  std::string name;
1028  std::string value;
1029  };
1030 
1031 The members are collected into one definition generated by soapcpp2. Members
1032 may be repeated from one definition to another, but only if their associated
1033 types are identical. So a third extension with a `value` member with a
1034 different type fails:
1035 
1036  mutable struct ns__tuple
1037  {
1038  duuble value; // BAD: value is already declared std::string
1039  };
1040 
1041 The `mutable` concept has proven to be very useful when declaring and
1042 collecting SOAP Headers for multiple services, which are collected into one
1043 `struct SOAP_ENV__Header` by the soapcpp2 tool.
1044 
1045 ### Default member values
1046 
1047 Class and struct data members may be declared with a default initialization
1048 value that is provided "inline" with the declaration of the member:
1049 
1050  class ns__record
1051  {
1052  public:
1053  std::string name = "Joe";
1054 
1055 These initializations are made by the default constructor that is added by
1056 soapcpp2 to each class and struct. A constructor is only added when a default
1057 constructor is not already defined with the class declaration.
1058 
1059 Initializations can only be provided for members that have primitive types
1060 (`bool`, `enum`, `time_t`, numeric and string types).
1061 
1062 ### Attribute members
1063 
1064 Class and struct data members can be declared as XML attributes by annotating
1065 their type with a `@` with the declaration of the member:
1066 
1067  class ns__record
1068  {
1069  public:
1070  @std::string name;
1071  @uint64_t SSN;
1072  ns__record *spouse;
1073  };
1074 
1075 This class maps to a complexType in the soapcpp2-generated schema:
1076 
1077  <complexType name="record">
1078  <sequence>
1079  <element name="spouse" type="ns:record" minOccurs="0" maxOccurs="1" nillable="true"/>
1080  </sequence>
1081  <attribute name="name" type="xsd:string" use="required"/>
1082  <attribute name="SSN" type="xsd:unsignedLong" use="required"/>
1083  </complexType>
1084 
1085 An example XML instance of `ns__record` is:
1086 
1087  <ns:record xmlns:ns="urn:types" name="Joe" SSN="1234567890">
1088  <spouse>
1089  <name>Jane</name>
1090  <SSN>1987654320</SSN>
1091  </spouse>
1092  </ns:record>
1093 
1094 Attribute data members are restricted to primitive types (`bool`, `enum`,
1095 `time_t`, numeric and string types), `xsd__hexBinary`, `xsd__base64Binary`, and
1096 custom serializers, such as `xsd__dateTime`. Custom serializers for types that
1097 may be used as attributes MUST define `soap_s2T` and `soap_T2s` functions that
1098 convert values of type `T` to strings and back.
1099 
1100 Attribute data members can be pointers and smart pointers to these types, which
1101 permits attributes to be optional.
1102 
1103 ### (Smart) pointer members and their occurrence constraints
1104 
1105 A public pointer-typed data member is serialized by following its (smart)
1106 pointer(s) to the value pointed to.
1107 
1108 Pointers that are NULL and smart pointers that are empty are serialized to
1109 produce omitted element and attribute values, unless an element is required
1110 and is nillable.
1111 
1112 To control the occurrence requirements of pointer-based data members,
1113 occurrence constraints are associated with data members in the form of a range
1114 `minOccurs : maxOccurs`. For non-repeatable (meaning, not a container or array)
1115 data members, there are only three reasonable occurrence constraints:
1116 
1117 - `0:0` means that this element or attribute is prohibited.
1118 - `0:1` means that this element or attribute is optional.
1119 - `1:1` means that this element or attribute is required.
1120 
1121 Pointer-based data members have a default `0:1` occurrence constraint, making
1122 them optional, and their XSD schema local element/attribute definition is
1123 marked as nillable. Non-pointer data members have a default `1:1` occurence
1124 constraint, making them required.
1125 
1126 A pointer data member that is explicitly marked as required with `1:1` will be
1127 serialized as an element with an xsi:nil attribute, thus effectively revealing
1128 the NULL property of its value.
1129 
1130 A non-pointer data member that is explicitly marked as optional with `0:1` will
1131 be set to its default value when no XML value is presented to the deserializer.
1132 A default value can be assigned to data members that have primitive types.
1133 
1134 Consider for example:
1135 
1136  class ns__record
1137  {
1138  public:
1139  std::shared_ptr<std::string> name; // optional (0:1)
1140  uint64_t SSN 0:1 = 999; // forced this to be optional with default 999
1141  ns__record *spouse 1:1; // forced this to be required (only married people)
1142  };
1143 
1144 This class maps to a complexType in the soapcpp2-generated schema:
1145 
1146  <complexType name="record">
1147  <sequence>
1148  <element name="name" type="xsd:string" minOccurs="0" maxOccurs="1" nillable="true"/>
1149  <element name="SSN" type="xsd:unsignedLong" minOccurs="0" maxOccurs="1" default="999"/>
1150  <element name="spouse" type="ns:record" minOccurs="1" maxOccurs="1" nillable="true"/>
1151  </sequence>
1152  </complexType>
1153 
1154 An example XML instance of `ns__record` with its `name` string value set to
1155 `Joe`, `SSN` set to its default, and `spouse` set to NULL:
1156 
1157  <ns:record xmlns:ns="urn:types">
1158  <name>Joe</name>
1159  <SSN>999</SSN>
1160  <spouse xsi:nil="true"/>
1161  </ns:record>
1162 
1163 ### Container members and their occurrence constraints
1164 
1165 Class and struct data members declared as a container `std::deque`, `std::list`,
1166 `std::set`, and `std::vector` are serialized as a collection of values:
1167 
1168  class ns__record
1169  {
1170  public:
1171  std::vector<std::string> names;
1172  uint64_t SSN;
1173  };
1174 
1175 To practically limit the number of names within reasonable bounds, occurrence
1176 constraints are associated with the container. Occurrence constraints are of
1177 the form `minOccurs : maxOccurs`:
1178 
1179  class ns__record
1180  {
1181  public:
1182  std::vector<std::string> names 1:10;
1183  uint64_t SSN;
1184  };
1185 
1186 This class maps to a complexType in the soapcpp2-generated schema:
1187 
1188  <complexType name="record">
1189  <sequence>
1190  <element name="name" type="xsd:string" minOccurs="1" maxOccurs="10"/>
1191  <element name="SSN" type="xsd:unsignedLong" minOccurs="1" maxOccurs="1""/>
1192  </sequence>
1193  </complexType>
1194 
1195 Because C does not support a container template library, we can use a
1196 dynamically-sized array of values. This array is declared as a size-pointer
1197 member pair:
1198 
1199  struct ns__record
1200  {
1201  $int sizeofnames; // array size
1202  char* *names; // array of char* names
1203  uint64_t SSN;
1204  };
1205 
1206 where the marker `$` with `int` denotes a special type that is used to store
1207 the array size and to indicate that this is a size-pointer member pair that
1208 declares a dynamically-sized array.
1209 
1210 This class maps to a complexType in the soapcpp2-generated schema:
1211 
1212  <complexType name="record">
1213  <sequence>
1214  <element name="name" type="xsd:string" minOccurs="0" maxOccurs="unbounded" nillable="true"/>
1215  <element name="SSN" type="xsd:unsignedLong" minOccurs="1" maxOccurs="1""/>
1216  </sequence>
1217  </complexType>
1218 
1219 To limit the number of names in the array within reasonable bounds, occurrence
1220 constraints are associated with the array size member. Occurrence constraints
1221 are of the form `minOccurs : maxOccurs`:
1222 
1223  struct ns__record
1224  {
1225  $int sizeofnames 1:10; // array size 1..10
1226  char* *names; // array of one to ten char* names
1227  uint64_t SSN;
1228  };
1229 
1230 This class maps to a complexType in the soapcpp2-generated schema:
1231 
1232  <complexType name="record">
1233  <sequence>
1234  <element name="name" type="xsd:string" minOccurs="1" maxOccurs="10" nillable="true"/>
1235  <element name="SSN" type="xsd:unsignedLong" minOccurs="1" maxOccurs="1""/>
1236  </sequence>
1237  </complexType>

1239 ### Union members
1240 
1241 A union member in a class or in a struct cannot be serialized unless a
1242 discriminating variant selector is provided that tells the serializer which
1243 union field to serialize.
1244 
1245 The variant selector is associated with the union as a selector-union member
1246 pair, where the variant selector is a special `$int` member:
1247 
1248  class ns__record
1249  {
1250  public:
1251  $int xORnORs; // variant selector
1252  union choice
1253  {
1254  float x;
1255  int n;
1256  char *s;
1257  } u;
1258  std::string name;
1259  };
1260 
1261 The variant selector values are auto-generated based on the union name `choice`
1262 and the names of its members `x`, `n`, and `s`:
1263 
1264 - `xORnORs = SOAP_UNION_choice_x` when `u.x` is valid.
1265 - `xORnORs = SOAP_UNION_choice_n` when `u.n` is valid.
1266 - `xORnORs = SOAP_UNION_choice_s` when `u.s` is valid.
1267 - `xORnORs = 0` when none are valid (should only be used with great care,
1268  because XML content validation may fail when content is required but absent).
1269 
1270 This class maps to a complexType with a sequence and choice in the
1271 soapcpp2-generated schema:
1272 
1273  <complexType name="record">
1274  <sequence>
1275  <choice>
1276  <element name="x" type="xsd:float" minOccurs="1" maxOccurs="1"/>
1277  <element name="n" type="xsd:int" minOccurs="1" maxOccurs="1"/>
1278  <element name="s" type="xsd:string" minOccurs="0" maxOccurs="1" nillable="true"/>
1279  </choice>
1280  <element name="names" type="xsd:string" minOccurs="1" maxOccurs="10" nillable="true"/>
1281  </sequence>
1282  </complexType>

1284 ### Adding get and set methods
1285 
1286 A public `get` method may be added to a class or struct, which will be
1287 triggered by the deserializer. This method will be invoked right after the
1288 instance is populated by the deserializer. The `get` method can be used to
1289 update or verify deserialized content. It should return `SOAP_OK` or set
1290 `soap::error` to a nonzero error code and return it.
1291 
1292 A public `set` method may be added to a class or struct, which will be
1293 triggered by the serializer. The method will be invoked just before the
1294 instance is serialized. Likewise, the `set` method should return `SOAP_OK` or
1295 set set `soap::error` to a nonzero error code and return it.
1296 
1297 For example, adding a `set` and `get` method to a class declaration:
1298 
1299  class ns__record
1300  {
1301  public:
1302  int set(struct soap*); // triggered before serialization
1303  int get(struct soap*); // triggered after deserialization
1304 
1305 To add these and othe rmethods to classes and structs with wsdl2h and
1306 `typemap.dat`, please see section [class and struct addition](#typemap3).

1309 
1310 To define and reference XML document root elements we use type names that start
1311 with an underscore:
1312 
1313  class _ns__record

1316 given type:
1317 
1318  typedef ns__record _ns__record;

1321 soapcpp2-generated schema:
1322 
1323  <element name="record" type="ns:record"/>
1324 
1325 An example XML instance of `_ns__record` is:
1326 
1327  <ns:record xmlns:ns="urn:types">
1328  <name>Joe</name>

1331  <name>Jane</name>

1333  </spouse>
1334  </ns:record>

1339 
1340  typedef std::string _ns__name 1:100;

1342  {
1343  public:
1344  @_QName xsi__type; // built-in XSD attribute xsi:type
1345  _ns__name ns__name; // ref to global ns:name element

1351

1355  <maxLength value="100"/>

1357  </simpleType>

1367  <element name="record" type="ns:record"/>

1370 not match their definitions:
1371 
1372  class _ns__record
1373  {
1374  public:
1375  int ns__name; // BAD: element ns:name is NOT of an int type
1376 
1377 Therefore, we recommend to avoid qualified member names and only use them when

1381 ### Operations on classes and structs
1382 
1383 The following functions/macros are generated by soapcpp2 for each type `T`,
1384 which should make it easier to send, receive, and copy XML data in C and in
1385 C++:
1386 
1387 - `int soap_write_T(struct soap*, T*)` writes an instance of `T` to a FILE (via
1388  `FILE *soap::sendfd)`) or to a stream (via `std::ostream *soap::os`).
1389  Returns `SOAP_OK` on success or an error code, also stored in `soap->error`.
1390 
1391 - `int soap_read_T(struct soap*, T*)` reads an instance of `T` from a FILE (via
1392  `FILE *soap::recvfd)`) or from a stream (via `std::istream *soap::is`).
1393  Returns `SOAP_OK` on success or an error code, also stored in `soap->error`.
1394 
1395 - `void soap_default_T(struct soap*, T*)` sets an instance `T` to its default
1396  value, resetting members of a struct to their initial values (for classes we
1397  use method `T::soap_default`, see below).
1398 
1399 - `T * soap_dup_T(struct soap*, T *dst, const T *src)` (soapcpp2 option `-Ec`)
1400  deep copy `src` into `dst`, replicating all deep cycles and shared pointers
1401  when a managing soap context is provided as argument. When `dst` is NULL,
1402  allocates space for `dst`. Deep copy is a tree when argument is NULL, but the
1403  presence of deep cycles will lead to non-termination. Use flag
1404  `SOAP_XML_TREE` with managing context to copy into a tree without cycles and
1405  pointers to shared objects. Returns `dst` (or allocated space when `dst` is
1406  NULL).
1407 
1408 - `void soap_del_T(const T*)` (soapcpp2 option `-Ed`) deletes all
1409  heap-allocated members of this object by deep deletion ONLY IF this object
1410  and all of its (deep) members are not managed by a soap context AND the deep
1411  structure is a tree (no cycles and co-referenced objects by way of multiple
1412  (non-smart) pointers pointing to the same data). Can be safely used after
1413  `soap_dup(NULL)` to delete the deep copy. Does not delete the object itself.
1414 
1415 When in C++ mode, soapcpp2 tool adds several methods to classes and structs, in
1416 addition to adding a default constructor and destructor (when these were not
1417 explicitly declared).
1418 
1419 The public methods added to a class/struct `T`:
1420 
1421 - `virtual int T::soap_type(void)` returns a unique type ID (`SOAP_TYPE_T`).
1422  This numeric ID can be used to distinguish base from derived instances.
1423 
1424 - `virtual void T::soap_default(struct soap*)` sets all data members to
1425  default values.
1426 
1427 - `virtual void T::soap_serialize(struct soap*) const` serializes object to
1428  prepare for SOAP 1.1/1.2 encoded output (or with `SOAP_XML_GRAPH`) by
1429  analyzing its (cyclic) structures.
1430 
1431 - `virtual int T::soap_put(struct soap*, const char *tag, const char *type) const`
1432  emits object in XML, compliant with SOAP 1.1 encoding style, return error
1433  code or `SOAP_OK`. Requires `soap_begin_send(soap)` and
1434  `soap_end_send(soap)`.
1435 
1436 - `virtual int T::soap_out(struct soap*, const char *tag, int id, const char *type) const`
1437  emits object in XML, with tag and optional id attribute and xsi:type, return
1438  error code or `SOAP_OK`. Requires `soap_begin_send(soap)` and
1439  `soap_end_send(soap)`.
1440 
1441 - `virtual void * T::soap_get(struct soap*, const char *tag, const char *type)`
1442  Get object from XML, compliant with SOAP 1.1 encoding style, return pointer
1443  to object or NULL on error. Requires `soap_begin_recv(soap)` and
1444  `soap_end_recv(soap)`.
1445 
1446 - `virtual void *soap_in(struct soap*, const char *tag, const char *type)`
1447  Get object from XML, with matching tag and type (NULL matches any tag and
1448  type), return pointer to object or NULL on error. Requires
1449  `soap_begin_recv(soap)` and `soap_end_recv(soap)`
1450 
1451 - `virtual T * T::soap_alloc(void) const` returns a new object of type `T`,
1452  default initialized and not managed by a soap context.
1453 
1454 - `virtual T * T::soap_dup(struct soap*) const` (soapcpp2 option `-Ec`) returns
1455  a duplicate of this object by deep copying, replicating all deep cycles and
1456  shared pointers when a managing soap context is provided as argument. Deep
1457  copy is a tree when argument is NULL, but the presence of deep cycles will
1458  lead to non-termination. Use flag `SOAP_XML_TREE` with managing context to
1459  copy into a tree without cycles and pointers to shared objects.
1460 
1461 - `virtual void T::soap_del() const` (soapcpp2 option `-Ed`) deletes all
1462  heap-allocated members of this object by deep deletion ONLY IF this object
1463  and all of its (deep) members are not managed by a soap context AND the deep
1464  structure is a tree (no cycles and co-referenced objects by way of multiple
1465  (non-smart) pointers pointing to the same data).Can be safely used after
1466  `soap_dup(NULL)` to delete the deep copy. Does not delete the object itself.

1470 
1471 A class or struct with the following layout is a one-dimensional SOAP Array
1472 type:

1475  {
1476  public:
1477  T *__ptr; // array pointer
1478  int __size; // array size
1479  };

1482

1488  };

1491 of `__size[0]*__size[1]* ... * __size[N-1]` elements.
1492

1494 xsd:base64Binary types when the pointer type is `unsigned char`:
1495 
1496  class xsd__hexBinary
1497  {
1498  public:
1499  unsigned char *__ptr; // points to raw binary data
1500  int __size; // size of data
1501  };
1502 
1503 and
1504 
1505  class xsd__base64Binary
1506  {
1507  public:
1508  unsigned char *__ptr; // points to raw binary data
1509  int __size; // size of data
1510  };
1511

1513 MIME/MTOM (and older DIME) attachments, such as in xop:Include elements:
1514 
1515  //gsoap xop schema import: http://www.w3.org/2004/08/xop/include
1516  class _xop__Include
1517  {

1523  char *options; // optional description of MIME attachment
1524  };

1527 user guide for more details.

1540 Memory management {#memory}
1541 =================
1542 
1543 Memory management with the `soap` context enables us to allocate data in
1544 context-managed heap space that can be collectively deleted. All deserialized
1545 data is placed on the context-managed heap by the gSOAP engine.
1546 
1547 Memory management in C {#memory1}
1548 ----------------------
1549 
1550 In C (wsdl2h option `-c` and soapcpp2 option `-c`), the gSOAP engine allocates
1551 data on a context-managed heap with:
1552 
1553 - `void *soap_malloc(struct soap*, size_t len)`.
1554 
1555 The `soap_malloc` function is a wrapper around `malloc`, but which also allows
1556 the `struct soap` context to track all heap allocations for collective deletion
1557 with `soap_end(soap)`:
1558 
1559  #include "soapH.h"
1560  #include "ns.nsmap"
1561  ...
1562  struct soap *soap = soap_new(); // new context
1563  ...
1564  struct ns__record *record = soap_malloc(soap, sizeof(struct ns__record));
1565  soap_default_ns__record(soap, record);
1566  ...
1567  soap_destroy(soap); // only for C++, see section on C++ below
1568  soap_end(soap); // delete record and all other heap allocations
1569  soap_free(soap); // delete context
1570 
1571 The soapcpp2 auto-generated deserializers in C use `soap_malloc` to allocate
1572 and populate deserialized structures, which are managed by the context for
1573 collective deletion.
1574 
1575 To make `char*` and `wchar_t*` string copies to the context-managed heap, we
1576 can use the functions:
1577 
1578 - `char *soap_strdup(struct soap*, const char*)` and
1579 
1580 - `wchar_t *soap_wstrdup(struct soap*, const wchar_t*)`.
1581 
1582 We use the soapcpp2 auto-generated `soap_dup_T` functions to duplicate data
1583 into another context (this requires soapcpp2 option `-Ec` to generate), here
1584 shown for C with the second argument `dst` NULL because we want to allocate a
1585 new managed structure:
1586 
1587  struct soap *other_soap = soap_new(); // another context
1588  struct ns__record *other_record = soap_dup_ns__record(other_soap, NULL, record);
1589  ...
1590  soap_destroy(other_soap); // only for C++, see section on C++ below
1591  soap_end(other_soap); // delete other_record and all of its deep data
1592  soap_free(other_soap); // delete context
1593 
1594 Note that the only reason to use another context and not to use the primary
1595 context is when the primary context must be destroyed together with all of the
1596 objects it manages while some of the objects must be kept alive. If the objects
1597 that are kept alive contain deep cycles then this is the only option we have,
1598 because deep copy with a managing context detects and preserves these
1599 cycles unless the `SOAP_XML_TREE` flag is used with the context:
1600 
1601  struct soap *other_soap = soap_new1(SOAP_XML_TREE); // another context
1602  struct ns__record *other_record = soap_dup_ns__record(other_soap, NULL, record);
1603 
1604 The resulting deep copy will be a full copy of the source data structure as a
1605 tree without co-referenced data (i.e. no digraph) and without cycles. Cycles
1606 are pruned and (one of the) pointers that forms a cycle is repaced by NULL.
1607 
1608 We can also deep copy into unmanaged space and use the auto-generated
1609 `soap_del_T()` function (requires soapcpp2 option `-Ed` to generate) to delete
1610 it later, but we MUST NOT do this for any data that we suspect has deep cycles:
1611 
1612  struct ns__record *other_record = soap_dup_ns__record(NULL, NULL, record);
1613  ...
1614  soap_del_ns__record(other_record); // deep delete record data members
1615  free(other_record); // delete the record
1616 
1617 Cycles in the data structure will lead to non-termination when making unmanaged
1618 deep copies. Consider for example:
1619 
1620  struct ns__record
1621  {
1622  const char *name;
1623  uint64_t SSN;
1624  ns__record *spouse;
1625  };
1626 
1627 Our code to populate a structure with a mutual spouse relationship:
1628 
1629  struct soap *soap = soap_new();
1630  ...
1631  struct ns__record pers1, pers2;
1632  soap_default_ns__record(soap, &pers1);
1633  soap_default_ns__record(soap, &pers2);
1634  pers1.name = "Joe"; // OK to serialize static data
1635  pers1.SSN = 1234567890;
1636  pers1.spouse = &pers2;
1637  pers2.name = soap_strdup(soap, "Jane"); // allocates and copies a string
1638  pers2.SSN = 1987654320;
1639  pers2.spouse = &pers1;
1640  ...
1641  struct ns__record *pers3 = soap_dup_ns__record(NULL, NULL, &pers1); // BAD
1642  struct ns__record *pers4 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1643  soap_set_mode(soap, SOAP_XML_TREE);
1644  struct ns__record *pers5 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1645 
1646 As we can see, the gSOAP serializer can serialize any heap, stack, or static
1647 allocated data, such as in our code above. So we can serialize the
1648 stack-allocated `pers1` record as follows:
1649 
1650  soap->sendfd = fopen("record.xml", "w");
1651  soap_set_mode(soap, SOAP_XML_GRAPH); // support id-ref w/o requiring SOAP
1652  soap_clr_mode(soap, SOAP_XML_TREE); // if set, clear
1653  soap_write_ns__record(soap, &pers1);
1654  fclose(soap->sendfd);
1655  soap->sendfd = NULL;
1656 
1657 which produces an XML document record.xml that is similar to:
1658 
1659  <ns:record xmlns:ns="urn:types" id="Joe">
1660  <name>Joe</name>
1661  <SSN>1234567890</SSN>
1662  <spouse id="Jane">
1663  <name>Jane</name>
1664  <SSN>1987654320</SSN>
1665  <spouse ref="#Joe"/>
1666  </spouse>
1667  </ns:record>
1668 
1669 Deserialization of an XML document with a SOAP 1.1/1.2 encoded id-ref graph
1670 leads to the same non-termination problem when we later try to copy the data
1671 into unmanaged space:
1672 
1673  struct soap *soap = soap_new1(SOAP_XML_GRAPH); // support id-ref w/o SOAP
1674  ...
1675  struct ns__record pers1;
1676  soap->recvfd = fopen("record.xml", "r");
1677  soap_read_ns__record(soap, &pers1);
1678  fclose(soap->recvfd);
1679  soap->recvfd = NULL;
1680  ...
1681  struct ns__record *pers3 = soap_dup_ns__record(NULL, NULL, &pers1); // BAD
1682  struct ns__record *pers4 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1683  soap_set_mode(soap, SOAP_XML_TREE);
1684  struct ns__record *pers5 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1685 
1686 Copying data with `soap_dup_T(soap)` into managed space is always safe. Copying
1687 into unmanaged space requires diligence. But deleting unmanaged data is easy
1688 with `soap_del_T()`.
1689 
1690 We can also use `soap_del_T()` to delete structures that we created in C, but
1691 only if these structures are created with `malloc` and do NOT contain pointers
1692 to stack and static data.
1693 
1694 Memory management in C++ {#memory2}
1695 ------------------------
1696 
1697 In C++, the gSOAP engine allocates data on a managed heap using a combination
1698 of `void *soap_malloc(struct soap*, size_t len)` and `soap_new_T()`, where `T`
1699 is the name of a class, struct, or class template (container or smart pointer).
1700 Heap allocation is tracked by the `struct soap` context for collective
1701 deletion with `soap_destroy(soap)` and `soap_end(soap)`.
1702 
1703 Only structs, classes, and class templates are allocated with `new` via
1704 `soap_new_T(struct soap*)` and mass-deleted with `soap_destroy(soap)`.
1705 
1706 There are four variations of `soap_new_T` for class/struct/template type `T`
1707 that soapcpp2 auto-generates to create instances on a context-managed heap:
1708 
1709 - `T * soap_new_T(struct soap*)` returns a new instance of `T` with default data
1710  member initializations that are set with the soapcpp2 auto-generated `void
1711  T::soap_default(struct soap*)` method), but ONLY IF the soapcpp2
1712  auto-generated default constructor is used that invokes `soap_default()` and
1713  was not replaced by a user-defined default constructor.
1714 
1715 - `T * soap_new_T(struct soap*, int n)` returns an array of `n` new instances of
1716  `T`. Similar to the above, instances are initialized.
1717 
1718 - `T * soap_new_req_T(struct soap*, ...)` returns a new instance of `T` and sets
1719  the required data members to the values specified in `...`. The required data
1720  members are those with minOccurs > 0, see the subsection on occurrence
1721  constraints in [Classes and structs](#toxsd9).
1722 
1723 - `T * soap_new_set_T(struct soap*, ...)` returns a new instance of `T` and sets
1724  the public/serializable data members to the values specified in `...`.
1725 
1726 The above functions can be invoked with a NULL `soap` context, but we will be
1727 responsible to use `delete T` to remove this instance from the unmanaged heap.
1728 
1729 Primitive types and arrays of these are allocated with `soap_malloc` by the
1730 gSOAP engine. As we stated above, all types except for classes, structs, class
1731 templates (containers and smart pointers) are allocated with `soap_malloc` for
1732 reasons of efficiency.
1733 
1734 We can use a C++ template to simplify the managed allocation and initialization
1735 of primitive values as follows (this is for primitive types only, because we
1736 should allocate structs and classes with `soap_new_T`):
1737 
1738  template<class T>
1739  T * soap_make(struct soap *soap, T val)
1740  {
1741  T *p = (T*)soap_malloc(soap, sizeof(T));
1742  if (p)
1743  *p = val;
1744  return p;
1745  }
1746 
1747 For example, assuming we have the following class:
1748 
1749  class ns__record
1750  {
1751  public:
1752  std::string name; // required name
1753  uint64_t *SSN; // optional SSN
1754  ns__record *spouse; // optional spouse
1755  };
1756 
1757 We can instantiate a record by using the auto-generated
1758 `soap_new_set_ns__record` and our `soap_make` to create a SSN value on the
1759 managed heap:
1760 
1761  soap *soap = soap_new(); // new context
1762  ...
1763  ns__record *record = soap_new_set_ns__record(
1764  soap,
1765  "Joe",
1766  soap_make<uint64_t>(soap, 1234567890),
1767  NULL);
1768  ...
1769  soap_destroy(soap); // delete record and all other managed instances
1770  soap_end(soap); // delete managed soap_malloc'ed heap data
1771  soap_free(soap); // delete context
1772 
1773 Note however that the gSOAP serializer can serialize any heap, stack, or static
1774 allocated data. So we can also create a new record as follows:
1775 
1776  uint64_t SSN = 1234567890;
1777  ns__record *record = soap_new_set_ns__record(soap, "Joe", &SSN, NULL);
1778 
1779 which will be fine to serialize this record as long as the local `SSN`
1780 stack-allocated value remains in scope when invoking the serializer and/or
1781 using `record`. It does not matter if `soap_destroy` and `soap_end` are called
1782 beyond the scope of `SSN`.
1783 
1784 To facilitate our class methods to access the managing context, we can add a
1785 soap context pointer to a class/struct:
1786 
1787  class ns__record
1788  {
1789  ...
1790  void create_more(); // needs a context to create more internal data
1791  protected:
1792  struct soap *soap; // the context that manages this instance, or NULL
1793  };
1794 
1795 The context is set when invoking `soap_new_T` (and similar) with a non-NULL
1796 context argument.
1797 
1798 We use the soapcpp2 auto-generated `soap_dup_T` functions to duplicate data
1799 into another context (this requires soapcpp2 option `-Ec` to generate), here
1800 shown for C++ with the second argument `dst` NULL because we want to allocate a
1801 new managed object:
1802 
1803  soap *other_soap = soap_new(); // another context
1804  ns__record *other_record = soap_dup_ns__record(other_soap, NULL, record);
1805  ...
1806  soap_destroy(other_soap); // delete record and other managed instances
1807  soap_end(other_soap); // delete other data (the SSNs on the heap)
1808  soap_free(other_soap); // delete context
1809 
1810 To duplicate base and derived instances when a base class pointer or reference
1811 is provided, use the auto-generated method `T * T::soap_dup(struct soap*)`:
1812 
1813  soap *other_soap = soap_new(); // another context
1814  ns__record *other_record = record->soap_dup(other_soap);
1815  ...
1816  soap_destroy(other_soap); // delete record and other managed instances
1817  soap_end(other_soap); // delete other data (the SSNs on the heap)
1818  soap_free(other_soap); // delete context
1819 
1820 Note that the only reason to use another context and not to use the primary
1821 context is when the primary context must be destroyed together with all of the
1822 objects it manages while some of the objects must be kept alive. If the objects
1823 that are kept alive contain deep cycles then this is the only option we have,
1824 because deep copy with a managing context detects and preserves these
1825 cycles unless the `SOAP_XML_TREE` flag is used with the context:
1826 
1827  soap *other_soap = soap_new1(SOAP_XML_TREE); // another context
1828  ns__record *other_record = record->soap_dup(other_soap); // deep tree copy
1829 
1830 The resulting deep copy will be a full copy of the source data structure as a
1831 tree without co-referenced data (i.e. no digraph) and without cycles. Cycles
1832 are pruned and (one of the) pointers that forms a cycle is repaced by NULL.
1833 
1834 We can also deep copy into unmanaged space and use the auto-generated
1835 `soap_del_T()` function or the `T::soap_del()` method (requires soapcpp2 option
1836 `-Ed` to generate) to delete it later, but we MUST NOT do this for any data
1837 that we suspect has deep cycles:
1838 
1839  ns__record *other_record = record->soap_dup(NULL);
1840  ...
1841  other_record->soap_del(); // deep delete record data members
1842  delete other_record; // delete the record
1843 
1844 Cycles in the data structure will lead to non-termination when making unmanaged
1845 deep copies. Consider for example:
1846 
1847  class ns__record
1848  {
1849  const char *name;
1850  uint64_t SSN;
1851  ns__record *spouse;
1852  };
1853 
1854 Our code to populate a structure with a mutual spouse relationship:
1855 
1856  soap *soap = soap_new();
1857  ...
1858  ns__record pers1, pers2;
1859  pers1.name = "Joe";
1860  pers1.SSN = 1234567890;
1861  pers1.spouse = &pers2;
1862  pers2.name = "Jane";
1863  pers2.SSN = 1987654320;
1864  pers2.spouse = &pers1;
1865  ...
1866  ns__record *pers3 = soap_dup_ns__record(NULL, NULL, &pers1); // BAD
1867  ns__record *pers4 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1868  soap_set_mode(soap, SOAP_XML_TREE);
1869  ns__record *pers5 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1870 
1871 Note that the gSOAP serializer can serialize any heap, stack, or static
1872 allocated data, such as in our code above. So we can serialize the
1873 stack-allocated `pers1` record as follows:
1874 
1875  soap->sendfd = fopen("record.xml", "w");
1876  soap_set_mode(soap, SOAP_XML_GRAPH); // support id-ref w/o requiring SOAP
1877  soap_clr_mode(soap, SOAP_XML_TREE); // if set, clear
1878  soap_write_ns__record(soap, &pers1);
1879  fclose(soap->sendfd);
1880  soap->sendfd = NULL;
1881 
1882 which produces an XML document record.xml that is similar to:
1883 
1884  <ns:record xmlns:ns="urn:types" id="Joe">
1885  <name>Joe</name>
1886  <SSN>1234567890</SSN>
1887  <spouse id="Jane">
1888  <name>Jane</name>
1889  <SSN>1987654320</SSN>
1890  <spouse ref="#Joe"/>
1891  </spouse>
1892  </ns:record>
1893 
1894 Deserialization of an XML document with a SOAP 1.1/1.2 encoded id-ref graph
1895 leads to the same non-termination problem when we later try to copy the data
1896 into unmanaged space:
1897 
1898  soap *soap = soap_new1(SOAP_XML_GRAPH); // support id-ref w/o SOAP
1899  ...
1900  ns__record pers1;
1901  soap->recvfd = fopen("record.xml", "r");
1902  soap_read_ns__record(soap, &pers1);
1903  fclose(soap->recvfd);
1904  soap->recvfd = NULL;
1905  ...
1906  ns__record *pers3 = soap_dup_ns__record(NULL, NULL, &pers1); // BAD
1907  ns__record *pers4 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1908  soap_set_mode(soap, SOAP_XML_TREE);
1909  ns__record *pers5 = soap_dup_ns__record(soap, NULL, &pers1); // OK
1910 
1911 Copying data with `soap_dup_T(soap)` into managed space is always safe. Copying
1912 into unmanaged space requires diligence. But deleting unmanaged data is easy
1913 with `soap_del_T()`.
1914 
1915 We can also use `soap_del_T()` to delete structures in C++, but only if these
1916 structures are created with `new` (and `new []` for arrays when applicable) for
1917 classes, structs, and class templates and with `malloc` for anything else, and
1918 the structures do NOT contain pointers to stack and static data.

1920 Features and limitations {#features}
1921 ========================
1922 
1923 There are several context initialization flags and mode flags to control XML
1924 serialization at runtime:
1925 
1926 - `SOAP_XML_STRICT`: strictly validates XML while deserializing. Should not be
1927  used together with SOAP 1.1/1.2 encoding style of messaging.
1928 
1929 - `SOAP_XML_INDENT`: produces indented XML.
1930 
1931 - `SOAP_XML_CANONICAL`: c14n canonocalization, removes unnecessary `xmlns`
1932  bindings and adds them to appropriate places by applying c14n normalization
1933  rules. Should not be used together with SOAP 1.1/1.2 encoding style of
1934  messaging.
1935 
1936 - `SOAP_XML_TREE`: write tree XML without id-ref, pruning data structure
1937  cycles to prevent nontermination.
1938 
1939 - `SOAP_XML_GRAPH`: write graph (digraph and cyclic graphs with shared pointers
1940  to objects) using id-ref attributes. That is, XML with SOAP multi-ref
1941  encoded id-ref elements. This is a structure-preserving serialization format,
1942  because co-referenced data and also cyclic relations are accurately represented.
1943 
1944 - `SOAP_XML_DEFAULTNS`: uses xmlns default bindings, assuming that the schema
1945  element form is "qualified" by default (be warned if it is not!).
1946 
1947 - `SOAP_XML_NOTYPE`: removes all xsi:type attribuation. This may affect the
1948  quality of the deserializer, which relies on xsi:type attributes to distinguish
1949  base class instances from derived class instanced.
1950 
1951 - `SOAP_C_UTFSTRING`: enables all `std::string` and `char*` strings to
1952  contain UTF-8 content.
1953 
1954 Additional notes with respect to the wsdl2h and soapcpp2 tools:
1955 
1956 - Nested classes, structs, and typedefs in a gSOAP header file are unnested by
1957  soapcpp2.
1958 
1959 - Use `#import "file.h"` instead of `#include` to import other header files in
1960  a gSOAP header file for soapcpp2. The `#include` and `#define` directives are
1961  accepted, but deferred to the generated code.
1962 
1963 - To remove any SOAP-specific bindings, use soapcpp2 option `-0`.
1964 
1965 - A gSOAP header file for soapcpp2 should not include any code statements, only
1966  data type declarations.
1967 
1968 - C++ namespaces are supported, use wsdl2h option `-qname`. Or add a `namespace
1969  name { ... }` to the header file, but the `{ ... }` MUST cover the entire
1970  header file content from begin to end.
1971 
1972 - Optional DOM support can be used to store mixed content or literal XML
1973  content. Otherwise, mixed content may be lost. Use wsdl2h option `-d` for
1974  DOM support and compile and link with `dom.c` or `dom.cpp`.

1977 Removing SOAP namespaces from XML payloads {#nsmap}
1978 ==========================================
1979 
1980 The soapcpp2 tool generates a `.nsmap` file that includes two bindings for SOAP
1981 namespaces. We can remove all SOAP namespaces (and SOAP processing logic) with
1982 soapcpp2 option `-0` or by simply setting the two entries to NULL:
1983 
1984  SOAP_NMAC struct Namespace namespaces[] =
1985  {
1986  {"SOAP-ENV", NULL, NULL, NULL},
1987  {"SOAP-ENC", NULL, NULL, NULL},
1988  ...
1989 
1990 Note that once the `.nsmap` is generated, we can copy-paste the content into
1991 our project code. However, if we rerun wsdl2h on updated WSDL/XSD files or
1992 `typemap.dat` declarations then we need to use the updated table.
